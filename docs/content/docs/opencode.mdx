title: OpenCode Integration
description: Route OpenCode traffic through Legroom for token compression, MCP tools, and cached model access. One command to wrap, one to unwrap.
---

Use `legroom wrap opencode` to route OpenCode LLM traffic through the Legroom proxy with a single command. The wrapper starts or reuses the proxy, writes OpenCode config, injects Legroom MCP tools, adds RTK context filtering, and launches OpenCode with the generated config.

The `legroom-opencode` npm package also exports a native OpenCode plugin. The plugin can be used directly from OpenCode config when you want in-process transport interception plus the Legroom retrieve tool.

## Quick Start

```bash
legroom wrap opencode
```

When you are done:

```bash
legroom unwrap opencode
```

## What `wrap opencode` Does

| Step | What happens |
|---|---|
| Proxy | Starts the Legroom proxy unless `--no-proxy` is set |
| Provider injection | Writes a `legroom` provider using `@ai-sdk/openai-compatible` into `opencode.json`, pointing at `http://127.0.0.1:<port>/v1` |
| Runtime env | Sets `OPENCODE_CONFIG_CONTENT` with provider, plugin, and optional local MCP config so OpenCode picks up Legroom at launch |
| Provider compatibility | Leaves `OPENAI_BASE_URL` and `ANTHROPIC_BASE_URL` untouched so OpenCode `/connect` providers keep their own routing |
| Context tool | Injects RTK (or `lean-ctx`) instructions into `~/.config/opencode/AGENTS.md` and project `AGENTS.md` |
| MCP setup | Registers the Legroom MCP server (`legroom_compress`, `legroom_retrieve`, `legroom_stats`) |
| Serena MCP | Optionally registers Serena code graph tools (`--no-serena` to skip) |
| Backup | Snapshots `opencode.json` to `opencode.json.legroom-backup` before making any changes |
| Launch | Starts the `opencode` binary through the proxy |

## Options

```bash
legroom wrap opencode \
  --port 8787 \
  --no-rtk \
  --no-mcp \
  --no-serena \
  --code-graph \
  --no-proxy \
  --learn \
  --memory \
  --backend anthropic \
  --anyllm-provider ... \
  --region ... \
  -- <opencode args>
```

## Provider Model Mapping

The generated `legroom` provider exposes these models through the proxy:

| Provider model | Upstream model |
|---|---|
| `legroom/claude-sonnet-4-6` | Claude Sonnet 4.6, 200K context, 16K output |
| `legroom/claude-opus-4-6` | Claude Opus 4.6, 200K context, 16K output |
| `legroom/claude-haiku-4-5-20251001` | Claude Haiku 4.5, 200K context, 8K output |
| `legroom/gpt-4o` | GPT-4o, 128K context, 16K output |
| `legroom/gpt-4.1` | GPT-4.1, 1M context, 32K output |

The default model is `legroom/claude-sonnet-4-6`. Change it in `opencode.json` or in the generated `OPENCODE_CONFIG_CONTENT` payload.

## Environment Variables

| Variable | Description |
|---|---|
| `OPENCODE_CONFIG_CONTENT` | JSON payload with provider, plugin, and optional local MCP config injected by `wrap` |
| `LEGROOM_PROXY_URL` | Proxy URL passed to Legroom MCP when a non-default port is used, and to the native plugin when configured |
| `LEGROOM_CONTEXT_TOOL` | Set to `lean-ctx` to use lean-ctx instead of RTK |

## Failure Learning

`legroom learn` supports OpenCode as a scan target. It reads past sessions from `~/.local/share/opencode/opencode.db` and writes corrections to your project's `AGENTS.md`.

```bash
legroom learn --agent opencode --apply
```

See [Failure Learning](/docs/failure-learning) for details on the learn system.

## Persistent Installs

`legroom install` supports OpenCode as a target for persistent provider wiring. Use provider scope when you want Legroom to edit `opencode.json` directly:

```bash
legroom install apply --preset persistent-service --scope provider --providers manual --target opencode
```

This writes the Legroom provider into `~/.config/opencode/opencode.json` and keeps the proxy running on port 8787.

The default user scope only writes shell environment configuration. For OpenCode, direct provider config requires `--scope provider`.

## Native OpenCode Plugin

The `legroom-opencode` package exports `LegroomPlugin` for direct OpenCode plugin registration. The plugin installs Legroom transport interception inside OpenCode, exposes the `legroom_retrieve` tool, and publishes Legroom metadata through the OpenCode plugin output env.

Example:

```ts
import { LegroomPlugin } from "legroom-opencode";

export default async function plugin(input) {
  return LegroomPlugin(input, {
    proxyUrl: process.env.LEGROOM_PROXY_URL ?? "http://127.0.0.1:8787",
  });
}
```

Use this plugin when OpenCode should intercept provider traffic in-process. Use `legroom wrap opencode` when you want the CLI to manage the proxy, config injection, MCP registration, backups, and unwrap behavior.

## Programmatic Config Helpers

The package also exports helpers for custom integrations:

```ts
import {
  buildOpencodeConfigContent,
  createLegroomProvider,
  createLegroomRetrieveTool,
} from "legroom-opencode";

const provider = createLegroomProvider({ proxyPort: 8787 });
const config = buildOpencodeConfigContent({
  proxyPort: 8787,
  defaultModel: "claude-sonnet-4-6",
});
const retrieve = createLegroomRetrieveTool({
  proxyBaseUrl: "http://127.0.0.1:8787",
});
```

## How It Works Under The Hood

1. **Config injection**. The wrapper writes a `provider.legroom` block into `opencode.json`. The provider uses `@ai-sdk/openai-compatible`, which OpenCode supports natively. Model mappings route requests through `http://127.0.0.1:<port>/v1`.
2. **Runtime config**. `OPENCODE_CONFIG_CONTENT` is set as an env var containing provider, plugin, and optional local MCP JSON. OpenCode reads it at startup and merges it with on-disk config.
3. **MCP tools**. Legroom registers `legroom_compress`, `legroom_retrieve`, and `legroom_stats` through `legroom mcp serve` unless `--no-mcp` is set.
4. **Native plugin path**. `LegroomPlugin` installs Legroom transport interception and uses `LEGROOM_PROXY_URL` or `http://127.0.0.1:8787` to reach the proxy.
5. **Unwrap**. `legroom unwrap opencode` restores `opencode.json` from the pre-wrap backup when present, strips Legroom marker blocks when no backup exists, and unregisters Legroom MCP servers.

## Troubleshooting

**OpenCode does not use the legroom provider.**

Check that `OPENCODE_CONFIG_CONTENT` is set and contains the `provider.legroom` block. The wrap command prints the env vars it sets.

**The native plugin cannot reach Legroom.**

Set `LEGROOM_PROXY_URL` to the running proxy URL, for example `http://127.0.0.1:8787`.

**Provider not found after unwrap.**

If unwrap left the provider configured, run `legroom unwrap opencode` again, or manually restore from `~/.config/opencode/opencode.json.legroom-backup`.

**Proxy port conflict.**

Use `--port` to select a specific port, or let the proxy auto-select an available one.
